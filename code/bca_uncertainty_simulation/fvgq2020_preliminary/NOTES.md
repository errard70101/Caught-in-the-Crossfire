## Solve-only adaptation

The original RED file mixes the model definition with MATLAB-side postprocessing
that depends on an external pruning toolbox. The local
`dynare/fvgq2020_solveonly.mod` removes that postprocessing and relies on
Dynare 6.2 native `pruning` plus MATLAB-side `simult_` calls.

Relative to `replicationRED_vendored.mod`, the solve-only adaptation:

1. Removes the inline `addpath(...)` call to the external pruning toolbox.
2. Removes the MATLAB block after the original `stoch_simul` calls.
3. Replaces the solve block with a Dynare 6.2 native
   `stoch_simul(order=3, pruning, k_order_solver, periods=0, irf=0, ...)`.
4. Replaces `resid(1)` with `resid;` for Dynare 6.x syntax.
5. Adds `set_dynare_seed(20260424);` for reproducible solving.
6. Adds four CKM prototype wedge observables (`wedge_A`, `wedge_G`,
   `wedge_l`, `wedge_x`) inside the Dynare model block.

Core FVGQ equilibrium equations, state definitions, and calibration values are
unchanged. The added CKM equations are accounting observables that do not feed
back into the FVGQ equilibrium system. Baseline `@#define higherphi = 0` is
preserved.

## Variable mapping to CKM inputs

| CKM input | FVGQ expression | Comment |
|-----------|-----------------|---------|
| `Y_t` | `y` | Aggregate output |
| `C_t` | `cspt` | `ifrac * ce + (1 - ifrac) * cw` |
| `I_t` | `ifrac * ivst` | Aggregate investment |
| `L_t` | `labor` | Aggregate hours |
| `K_t` | `k` | Aggregate capital |

The uncertainty shocks of interest are:

- `uA`: TFP stochastic-volatility innovation.
- `ue`: collateral-LTV stochastic-volatility innovation.

Ambient noise sources retained in simulation but not targeted for identification:

- `ud`: preference stochastic-volatility innovation.
- `ethet`, `eA`, `ed`: level shocks.

## Calibration provenance

The preliminary pipeline uses the exact FVGQ calibration in the vendored RED
file. The CKM prototype later maps those aggregates into a representative-agent
accounting system with:

- `alpha = 0.36`
- `beta = 0.994`
- `delta = 0.01625`
- `gamma = 2`
- `nu = 1`
- `chi` matched to FVGQ steady-state labor

This is a provisional accounting translation layer, not a Taiwan calibration.

## 2026-04-25 â€” Migration to Inside-Dynare wedge extraction

The preliminary pipeline now treats the CKM accounting wedges as true DGP
observables generated by `dynare/fvgq2020_solveonly.mod`.

Implementation details:

1. `fvgq2020_solveonly.mod` declares `ckm_gamma = 2`, `ckm_nu = 1`, and
   `ckm_chi` calibrated to the FVGQ steady state, then appends four auxiliary
   accounting identities:
   - `wedge_A = log(y) - alpha log(k(-1)) - (1-alpha) log(labor)`
   - `wedge_G = 1 - cspt/y - ifrac*ivst/y`
   - `wedge_l = 1 - chi L^(nu+1) C^gamma / ((1-alpha)Y)`
   - the CKM Euler equation with `wedge_x(+1)` in the expectation term.
2. `run_bca_analysis.m` writes `wedges_exp1.csv` by renaming the Inside-Dynare
   columns to the downstream convention: `log_A`, `G_share`, `tau_l`, `tau_x`.
3. `estimate_ckm_wedges_likelihood.m` no longer calls a MATLAB-side extractor.
   Its VAR(1) transition initialization uses
   `[sim_table.wedge_A, sim_table.wedge_l, sim_table.wedge_x]`, and `G_share`
   is copied from `sim_table.wedge_G`.
4. GIRF and Monte Carlo LP code now read wedges directly from simulated Dynare
   path matrices. The old algebraic fixed-point extractor has been removed.
5. Plots and comparison labels now use **True DGP (Inside-Dynare)** versus
   **Likelihood (Kalman filter)**.

This migration eliminates the standard algebraic fixed-point forecast error in
the investment wedge: the forward-looking accounting identity is solved by
Dynare as part of the model solution instead of being approximated ex post over
a finite simulated path.
